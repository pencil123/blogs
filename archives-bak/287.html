CPU 平均负载
admin 2015/09/16 2018-04-29

CPU load average 称为CPU 平均负载用来评估CPU的运行状态。不过其值更确切的是说明“目前CPU运行或运行队列的长度”。

Linux 平均负载

所有Unix和类Unix系统在内核中生成三个“平均负载”数量的无量纲度量。用户可以通过运行uptime命令(w命令或top命令)轻松地从Unix shell查询当前结果。
在GNU / Linux中，它们也可以通过读取/proc/loadavg文件来访问。

无量纲度量

在量纲分析中，无量纲量，或称无因次量、无维量、无维度量、无维数量、无次元量等，指的是没有量纲的量。它是个单纯的数字，量纲为1。无量纲量在数学、物理学、工程学、经济学以及日常生活中（如数数）被广泛使用。一些广为人知的无量纲量包括圆周率（π）、欧拉常数（e）和黄金分割率（φ）等。与之相对的是有量纲量，拥有诸如长度、面积、时间等单位。

平均负载计算方式

一台空闲的计算机的负载数为0（空闲进程不计算在内）。
每个使用或等待CPU的进程（就绪队列或运行队列）将Load 值增1；终止的每个进程将其减1。
大多数UNIX系统仅计算运行（在CPU上）或可运行（等待CPU）状态下的进程。但是，Linux还包含处于不间断睡眠状态（通常等待磁盘活动）的进程，如果由于I/O系统繁忙或停顿导致许多进程在I/O中被阻塞，则可能导致明显不同的结果。
例如，这包括由于NFS服务器故障或者速度太慢的介质（例如，USB 1.x存储设备）而导致的进程阻塞。这种情况可能会导致负载平均值升高，并不能反映CPU使用的实际增加（但仍可以了解用户需要等待多长时间）。

数值受历史影响

系统将负载平均值计算为负载数的指数衰减/加权移动平均值。负载平均值的三个数值是指系统运行的过去一分钟，五分钟和十五分钟。从数学上讲，自系统启动以来，所有三个值总是平均所有系统负载。它们都呈指数衰减，但它们以不同的速度衰减：它们分别在1分钟，5分钟和15分钟后以e指数衰减。
因此，1分钟的平均负荷包括最后一分钟负荷的63％（更准确地说：1-1 / e）和自启动以来的平均负荷的37％（1 / e），不包括最后一分钟。对于5分钟和15分钟的负载平均值，分别在5分钟和15分钟内计算相同的63％/ 37％比率。
因此，1分钟的平均载荷仅包括最近60秒的活动在技术上并不准确，因为它包括过去37％的活动，但说明它主要包括最后一分钟是正确的。

单CPU和多CPU不同

对于受CPU限制的单CPU系统，可以将负载平均值视为相应时间段内系统利用率的度量。对于具有多个CPU的系统，必须将负载除以处理器的数量以获得可比较的度量。

单CPU

例如，可以解释单CPU系统上的平均负载为“1.73 0.60 7.98”，因为：
在最后一分钟，系统平均负载超过73％（1.73可运行进程，因此0.73进程必须等待平均一个CPU系统）。
在过去的5分钟内，CPU平均怠速40％。
在过去的15分钟内，系统平均负载超过698％（7.98个可运行进程，因此6.98个进程平均需要等待单个CPU系统的运行）。
这意味着如果系统（CPU，磁盘，内存等）的速度是1.73倍，那么它可以处理最后一分钟的所有工作。

多CPU

在具有四个CPU的系统中，平均3.73的平均负载将表明平均有3.73个进程准备运行，并且每个进程都可以调度到CPU中。

确定队列长度

在现代UNIX系统上，线程相对于加载平均值的处理方式各不相同。
一些系统将线程视为进程以达到平均负载计算的目的：
等待运行的每个线程将向负载加1。
但是，其他系统，特别是实现所谓M：N线程的系统，使用不同的策略，
例如只为加载目的计算一次进程（不管线程数量多少），还是只计算当前用户暴露的线程 – 线程调度程序到内核，这可能取决于进程上设置的并发级别。Linux似乎将每个线程单独计数为加载1

补充

CPU 平均负载只能看出CPU运行或可运行队列的长度；并不能很好的判断CPU的状态。
数据监控中cacti是直接采用“load average”进行绘图，而zabbix采用‘load average’/core number 即per core 平均负载。
