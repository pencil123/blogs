Http Host 主机头攻击
admin 2018/05/02 2018-05-02

在web访问过程中，客户端可以任意修改request header fields。这样我们就要明确一点，所有来自客户端的数据信息都不可信。
当客户端在请求头中人为添加了host头字段信息。这种就是利用http协议修改Host请求头攻击行为。

生效过程

客户端访问http://example.com/时，添加request header 中host=evil.com。
这种情况，会请求到example.com的服务器上；服务器会认为客户端访问的域名为evil.com。
如果web服务上没有配置evil.com这个域名，web服务会使用默认配置虚拟主机响应此请求。
这就是漏洞所在。

密码重置

当web服务(nginx\apache\Varnish)收到host参数，它不能确认这个值是否被客户端上传host header field参数覆盖；会传输给后端的应用程序。
当后端的应用程序用到HOST这个数值时，就有可能会中招。
如：
<link href="http://_SERVER['HOST']" (Joomla)
并追加秘密密钥和令牌给包含它的链接：
<a href="http://_SERVER['HOST']?token=topsecret"> (Django, Gallery, others)
甚至直接从它导入脚本：
<script src="http://_SERVER['HOST']/misc/jquery.js?v=1.4.4"> (Various)
密码重置就是上述技术过程的一种实现方式。
如果后端应用程序的密码重置机制是发送验证邮件，对用户生成特定的key值，将host域名和key值拼接为特定的URL地址用于验证信任。
如果host域名获取错误，用户点击此URL就会将key值发送到恶意网站上去。黑客依此得到用户的key值，就可以重置用户的密码。

攻击方式实施过程

黑客打开目标网站申请重置特定用户的密码，如果后端应用程序存在这样的BUG。黑客就可以重置用户密码。

页面缓存中毒

WEB服务器端中毒

WEB服务一般都可以设置页面缓存的功能；其页面缓存内容会和访问的域名进行绑定。目前常用的WEB服务器：
Varnish使用它看到的第一个主机头来识别请求。Apache将所有存在的主机头连接起来。Nginx使用最后一个主机头。
这意味着您可以通过发出以下格式的请求来使Varnish缓存中毒。
> GET / HTTP/1.1
> Host: example.com
> Host: evil.com
当服务器使用Varnish服务做请求缓存时，上面的请求如果和example.com请求页面展示内容不同；就会污染example.com的缓存。

应用程序级缓存中毒

Joomla将Host头部写入每个页面，而不对其进行HTML编码，并且其缓存完全无视Host头部。在Joomla安装的主页上获得持久的XSS非常简单：
curl -H "Host: cow\"onerror='alert(1)'rel='stylesheet'" http://example.com/ | fgrep cow\"
这将创建以下请求：
> GET / HTTP/1.1
> Host: cow"onerror='alert(1)'rel='stylesheet'
响应应该显示一个有毒的<link>元素：
<link href="http://cow"onerror='alert(1)'rel='stylesheet'/" rel="canonical"/>
要验证缓存是否中毒，只需在浏览器中加载主页并观察弹出窗口。

保护服务器安全

1、设置默认虚拟主机
http host 主机头攻击，一般都是web服务器中的默认虚拟主机进行响应请求。在web服务器上设置默认的虚拟主机，并设置用来不做响应就可以解决这种问题。

2、程序中尽量不要使用_SERVER[‘HOST’]
程序中尽量不要使用此参数或者做好域名验证工作。并且注意黑客可能会传递host参数为SQL注入语句。
