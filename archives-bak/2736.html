Gearman 任务管理
admin 2017/11/24 2018-06-05

Gearman 服务是用来管理任务的服务；可以接受任务提交，并将任务分发给工作进程。

特点说明

1、分布式；工作进程可以在不同的服务器上，从Gearmand获取任务。
2、跨语言；gearman 服务扩展了多种语言的支持。client 和 worker 可以使用不同开发语言实现。
3、开源、灵活、上手快、易植入

架构和工作原理

Gearman 使用由三部分组成：一个Client、一个Worker、一个任务服务器。
1、Client提交一个 Job 任务交给 Job Server 任务服务器。它是请求的发起者。
2、Job Server 会去寻找一个合适的 Worker 来完成这项任务。任务调度者
3、Worker 执行由 Client 发送过来的 Job，并且将结果通过 Job Server 返回给 Client。
Gearman 提供了 Client 和 Worker 的 API，利用这些API 应用可以同 Gearman Job Server来进行通信。Gearman 内部 Client 和 Worker 之间的通信都是通过 TCP 连接来进行的。

Gearmand命令

-L [–listen] arg 地址服务器应该监听。缺省值是INADDR_ANY。
-P [–pid-file] arg（= / var / gearmand.pid） 将进程ID写入的文件。
-b [–backlog] arg（= 32） listen监听端口维持的半连接数。
-d [ –daemon] 守护进程，分离并在后台运行。
-f [–file-descriptors] arg 允许进程的文件描述符的数量（总连接数将稍小）。默认值是用户最多允许的。
-j [–job-retries] arg（= 0） 在作业服务器删除作业之前尝试运行作业的次数。这有助于确保不好的工作不会使所有可用的工作人员崩溃。默认是没有限制的。
–job-handle-prefix arg 用于生成作业处理字符串的前缀。如果未提供，则使用默认的“H：<主机名>”。
–hashtable-bucket arg（= 991） 内部作业哈希表中的桶数。对于大约三百万个队列中的工作，缺省值为991。如果队列中的任务数量随时超过三百万，则按比例使用较大的值（991 *工作数/ 3M）。
–keepalive 在套接字上启用keepalive。
–keepalive-idle arg（= -1） 如果启用keepalive，则为支持它的系统设置TCP_KEEPIDLE的值。值为-1意味着系统不支持它或尝试检索默认值时发生错误。
–keepalive-interval arg（= -1） 如果启用keepalive，则为支持它的系统设置TCP_KEEPINTVL的值。值为-1意味着系统不支持它或尝试检索默认值时发生错误。
–keepalive-count arg（= -1） 如果启用了keepalive，请为支持它的系统设置TCP_KEEPCNT的值。值为-1意味着系统不支持它或尝试检索默认值时发生错误。
-l [–log-file] arg（= / var / log / gearmand.log） 将错误和信息写入日志文件。如果log-file参数指定为“stderr”，则输出将转到stderr。如果“无”，则不会生成日志文件。
–verbose arg（= ERROR） 设置详细级别（FATAL, ALERT, CRITICAL, ERROR, WARNING, NOTICE, INFO, DEBUG）。
-r [–protocol] arg 加载协议模块。
-R [ – round-robin] 按照每个工作人员连接的循环顺序分配工作。默认是按工作人员添加的功能顺序分配工作。 -q [–queue-type] arg（= builtin）要使用的持久化队列类型。
–config-file arg（= / etc / gearmand.conf） 也可以用’@name’来指定
–syslog 使用syslog。
–coredump 是否为未被捕获的信号创建核心转储。
-t [–threads] arg（= 4） 要使用的I / O线程数量，0意味着gearmand将尝试猜测可以使用的最大数量。默认值= 4。
-u [–user] arg 启动后切换到给定的用户。
-V [–version] 显示gearmand的版本并退出。
-w [–worker-wakeup] arg（= 0） 为每个收到的作业唤醒的工作人员数量。默认是唤醒所有可用的工作人员。
–exceptions 默认情况下启用协议例外。
-h [–help] 打印此帮助菜单。

Gearman命令

Common options to both client and worker modes.

-f – Function name to use for jobs (can give many)处理任务的函数名
-h – Job server host （Job Server主机，默认是localhost）
-H – Print this help menu
-p – Job server port （Job Server端口，默认是4730）
-t – Timeout in milliseconds （执行多长时间超时，微秒）
-i – Create a pidfile for the process （创建进程的pid文件）

Client部分参数

-b – Run jobs in the background （后台运行任务）
-I – Run jobs as high priority （高优先级运行任务）
-L – Run jobs as low priority （低优先级运行任务）
-n – Run one job per line （逐行执行任务）
-N – Same as -n, but strip off the newline
-P – Prefix all output lines with functions names （在输入结果前面加处理的函数名）
-s – Send job without reading from standard input 执行任务不返回结果
-u – Unique key to use for job 任务的唯一标识

Worker部分参数

-c – Number of jobs for worker to run before exiting （统计worker进程处理多少个任务后中止）
-n – Send data packet for each line
-N – Same as -n, but strip off the newline
-w – Run in worker mode 以worker模式运行

Gearadmin 命令

-h [ –host ] arg (=localhost) 要连接的 host ，默认为「localhost」
-p [ –port ] arg (=4730) 要连接到的端口号，默认为 4730
–server-version 获得 gearman 服务器的版本号
–server-verbose 获得服务器的详细设置
–create-function arg 从服务器创建一个函数
–cancel-job arg 从服务器中取消置顶的任务
–drop-function arg 从服务器中删除一个函数
–show-unique-jobs 列出服务器上不重复的任务
–show-jobs 列出服务器上所有的任务
–getpid 获取服务器的进程 ID
–status 服务器的状态
–workers 服务器的 workers
–shutdown 关闭服务器
-S [ –ssl ] 启用 SSL 链接的支持

持久化

gearman的job server中的工作队列存储在内存中，一旦服务器有未处理的任务时重启或者宕机，那么这些任务就会丢失。
持久化存储队列可以允许添加后台任务，并将其存储在外部的持久型队列里。支持mysql，pgsql,sqlite,libtokyocabinet。

官方文档：http://gearman.org/manual/job_server/#persistent_queues

1、手动创建存储数据的表
CREATE DATABASE gearman;
CREATE TABLE `gearman_queue` (
`unique_key` varchar(64) NOT NULL,
`function_name` varchar(255) NOT NULL,
`priority` int(11) NOT NULL,
`data` longblob NOT NULL,
`when_to_run` int(11),
PRIMARY KEY (`unique_key`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8;

2、启动gearmand时，指定存储信息
gearmand -q mysql \
--mysql-host=192.168.1.100 \
--mysql-port=3306 \
--mysql-user=gearman \
--mysql-password=123456 \
--mysql-db=gearman \
--mysql-table=gearman_queue
