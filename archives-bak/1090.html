tornado ioloop 运行过程
admin 2016/12/28 2016-12-29

简单梳理一下 IOloop 继承了Configurable类(另外tornado.httpserver.HTTPServer也继承此；不谈)。
其中使用__new__函数使IOloop(Configurable)类的单实例;
def __new__(cls, *args, **kwargs):
base = cls.configurable_base()
~~~~init_kwargs = {}
++++print(“Configurable class __new__ method :cls is %s; base is %s” % (cls,base))
~~~~if cls is base:
~~~~~~~~impl = cls.configured_class()
++++++++print(“impl is %s” % (impl))
~~~~~~~~if base.__impl_kwargs:
~~~~~~~~~~~~init_kwargs.update(base.__impl_kwargs)
~~~~else:
~~~~~~~~impl = cls
~~~~init_kwargs.update(kwargs)
~~~~instance = super(Configurable, cls).__new__(impl)
++++print(“instance is %s” % (instance))~~~~
~~~~# initialize vs __init__ chosen for compatibility with AsyncHTTPClient
~~~~# singleton magic. If we get rid of that we can switch to __init__
~~~~# here too.
~~~~instance.initialize(*args, **init_kwargs)
++++print(“instance is %s” % (instance))
~~~~return instance
~~~~
output:
Configurable class __new__ method :
cls is <class ‘tornado.ioloop.IOLoop’>; base is <class ‘tornado.ioloop.IOLoop’>
impl is <class ‘tornado.platform.epoll.EPollIOLoop’>
instance is <tornado.platform.epoll.EPollIOLoop object at 0x7f318f415150>

从上可以知道：
impl 是EPollIOLoop的类；
__new__中的super调用Configurable的父类（也就是object）实例化一个impl(也就是EPollIOLoop)的对象。并不存在Configurable和ioloop的对象。

impl对象 的类是EPollIOLoop  –>父类为 PollIOLoop  —>父类为 IOLoop –>父类为Configurable

所有的http请求都会由对象中PollIOLoop类中定义的start函数执行中的while True的死循环处理的；

请求处理的过程：
1、IOLoop class add_timeout|add_callback method 会将请求提交给对象中EPollIOLoop类中的call_at; deadline默认now；callback 默认function timeout_callback|null_wrapper；
1.1、如果请求处理中存在tornado.ioloop.IOLoop.instance().add_timeout(time.time() + 15, callback=self.on_response)；则deadline 为15秒以后，callback 为self.on_response
1.2、callback函数会被IOLoop中的_run_callback(self,callback)函数调用执行；
2、call_at 函数会实例化_Timeout类；并将对象写入self._timeouts 堆中
2.1、 self_timeouts 是一个小顶堆（最小堆）结构，按照超时时间从小到大排列的 fd 的任务堆（ 通常这个任务都会包含一个 callback ）
3、然后就是在start 里面的死循环中的执行处理了~
