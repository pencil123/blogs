INFORMATION_SCHEMA提供对数据库元数据的访问 ，有关MySQL服务器的信息（例如数据库或表的名称），列的数据类型或访问权限。有时用于此信息的其他术语是 数据字典和 系统目录。

INFORMATION_SCHEMA使用说明

INFORMATION_SCHEMA是每个MySQL实例中的一个数据库，它存储有关MySQL服务器维护的所有其他数据库的信息。该 INFORMATION_SCHEMA数据库包含多个只读表。它们实际上是视图，而不是基表，因此没有与它们关联的文件，并且您无法在它们上设置触发器。此外，没有该名称的数据库目录。

虽然你可以选择INFORMATION_SCHEMA与一个默认的数据库USE 语句，你只能读取表的内容，不执行 INSERT， UPDATE或 DELETE对他们的操作。

INFORMATION_SCHEMA作为SHOW语句的替代品

该SELECT ... FROM INFORMATION_SCHEMA声明的目的是作为一个更加一致的方式来提供访问由各种提供的资料 SHOW陈述，MySQL支持（SHOW DATABASES， SHOW TABLES，等）。SELECT与以下相比，使用 具有以下优点SHOW：

  • 它符合Codd的规则，因为所有的访问都是在表上完成的。

  • 您可以使用熟悉的SELECT语句语法 ，只需要学习一些表名和列名。

  • 实现者无需担心添加关键字。

  • 您可以筛选，排序，连接并将INFORMATION_SCHEMA查询结果转换为您的应用程序需要的任何格式，例如数据结构或解析的文本表示。

  • 这种技术与其他数据库系统更具互操作性。例如，Oracle数据库用户熟悉查询Oracle数据字典中的表。

由于SHOW熟悉和广泛使用，这些SHOW陈述仍然是一种替代选择。事实上，实施沿 INFORMATION_SCHEMA，有增强 SHOW在描述 第24.34，“额外资讯显示陈述”。

特权

每个MySQL用户都有权访问这些表，但只能看到表中与用户具有适当访问权限的对象相对应的行。在某些情况下（例如表中的ROUTINE_DEFINITION列 INFORMATION_SCHEMA.ROUTINES），没有足够权限的用户可以看到NULL。这些限制不适用于 InnoDB表格; 你只能看到他们的PROCESS特权。

INFORMATION_SCHEMA通过SHOW声明 选择信息和查看相同信息的权限 相同。在任何一种情况下，您都必须拥有某个对象的特权才能看到关于它的信息。

性能考虑

INFORMATION_SCHEMA从多个数据库搜索信息的查询可能需要很长时间并影响性能。要检查查询的效率，可以使用 EXPLAIN。有关使用 EXPLAIN输出来调整 INFORMATION_SCHEMA查询的信息，请参见 第8.2.3节“优化INFORMATION_SCHEMA查询”。

标准考虑

为实现INFORMATION_SCHEMA 在MySQL表结构遵循ANSI / ISO SQL：2003标准的第11部分图式。我们的意图是近似符合SQL：2003核心功能F021 基本信息模式。

SQL Server 2000用户（也遵循标准）可能会注意到很强的相似性。但是，MySQL忽略了许多与我们的实现无关的列，并添加了特定于MySQL的列。一个这样的列是 表中的ENGINE 列INFORMATION_SCHEMA.TABLES。

尽管其他DBMS使用各种名称，例如 syscat或system，标准名称是INFORMATION_SCHEMA。

为避免使用标准或DB2，SQL Server或Oracle中保留的任何名称，我们更改了标记为“ MySQL扩展 ”的某些列的名称 。（例如，我们改变了 COLLATION对TABLE_COLLATION 在TABLES表中。）查看附近的这篇文章的结尾的保留字列表： https://web.archive.org/web/20070428032454/http://www.dbazine.com/db2 / db2-disarticles / gulutzan5。

    如果事件的时间由 EVERY子句而不是 AT子句决定（即，如果事件正在重复发生），则此列的值为 NULL。

  • INTERVAL_VALUE：对于重复发生的事件，此列包含事件EVERY子句的数字部分 。

    对于一次性事件（即其时序由AT子句确定的事件），此 列为NULL。

  • INTERVAL_FIELD：对于重复发生的事件，此列包含EVERY控制事件发生时间的子句的单位部分 。因此，此列包含诸如“ YEAR'，' QUARTER'，' DAY'等的值。

    对于一次性事件（即其时序由AT子句确定的事件），此 列为NULL。

  • SQL_MODE：创建或更改事件并在其下执行事件时有效的SQL模式。有关允许的值，请参见 第5.1.8节“服务器SQL模式”。

  • STARTS：对于周期性活动，其定义包括STARTS子句，该列包含对应的 DATETIME值。与 EXECUTE_AT列一样，此值可解析所使用的任何表达式。

    如果没有STARTS影响事件发生时间的条款，则此栏为NULL

  • ENDS：对于周期性活动，其定义包括ENDS子句，该列包含对应的 DATETIME值。与 EXECUTE_AT列一样，此值可解析所使用的任何表达式。

    如果没有ENDS影响事件发生时间的条款，则此栏为NULL。

  • STATUS：其中的三个值 ENABLED，DISABLED或 SLAVESIDE_DISABLED。

    SLAVESIDE_DISABLED表示事件的创建发生在另一台作为复制主服务器的MySQL服务器上，并被复制到当前作为从服务器的MySQL服务器上，但事件目前并未在从服务器上执行。有关更多信息，请参见 第16.4.1.16节“调用的功能的复制”。

  • ON_COMPLETION：两个值之一 PRESERVE或NOT PRESERVE。

  • CREATED：事件创建的日期和时间。这是一个 TIMESTAMP价值。

  • LAST_ALTERED：事件最后修改的日期和时间。这是一个 TIMESTAMP价值。如果该事件自创建以来未被修改，则此列与列保持相同的值CREATED。

  • LAST_EXECUTED：事件上次执行的日期和时间。一个DATETIME 值。如果事件从未执行过，则此列为 NULL。

    LAST_EXECUTED指示事件何时开始。因此，该ENDS列永远不会少于LAST_EXECUTED。

  • EVENT_COMMENT：评论的文本，如果事件有一个。如果不是，则此列的值为空字符串。

  • ORIGINATOR：创建事件的MySQL服务器的服务器ID; 用于复制。默认值是0。

  • CHARACTER_SET_CLIENT：character_set_client 创建事件时系统变量的会话值。

  • COLLATION_CONNECTION：collation_connection 创建事件时系统变量的会话值。

  • DATABASE_COLLATION：与事件关联的数据库的归类。

MySQL的> SELECT VIEW_DEFINITION FROM INFORMATION_SCHEMA.VIEWS

+ ---------------------------------- +

+ ---------------------------------- +
一排（0.00秒）

这些表格中的行代表快照。在这种情况下 TP_THREAD_STATE，一个线程组的所有行包含一个快照。因此，MySQL服务器在生成快照时保存线程组的互斥体。但它并不同时在所有线程组上保留互斥锁，以防止TP_THREAD_STATE阻止整个MySQL服务器的声明。

  • TP_GROUP_ID

    线程组ID。这是表格中的唯一键。

    消费者线程的数量。如果活动线程被阻塞或阻塞，最多只有一个线程可以开始执行。

    保留状态中的线程数量。这意味着只有在需要唤醒新线程并且没有消费者线程时才会启动它们。当线程组创建了比正常操作所需更多的线程时，这是大多数线程结束的地方。通常一个线程组需要额外的线程一段时间，然后不再需要它们一段时间。在这种情况下，他们进入保留状态并保持直到需要再次使用。他们占用一些额外的内存资源，但没有额外的计算资源。

  • CONNECT_THREAD_COUNT

    正在处理或等待处理连接初始化和身份验证的线程数。每个线程组最多可以有四个连接线程; 这些线程在一段时间不活动后过期。

    本专栏是在MySQL 5.7.18中添加的。

    使用此线程组的连接数。

  • QUEUED_QUERIES

    在高优先级队列中等待的语句的数量。

  • QUEUED_TRANSACTIONS

    在低优先级队列中等待的语句的数量。这些是尚未开始的事务的初始语句，因此它们也表示排队的事务。

  • ACTIVE_THREAD_COUNT

    活动执行语句的线程数。

    执行的语句数量。这个数字在语句开始执行时递增，而不是在语句结束时递增。

  • QUERIES_QUEUED

    收到的排队等待执行的语句数量。这不会计算线程组可以在没有排队的情况下立即开始执行的语句，这可能在5.5.3.3节“线程池操作”中描述的条件下发生 。

  • THREADS_STARTED

    线程数已开始。

  • PRIO_KICKUPS

    根据thread_pool_prio_kickup_timer 系统变量的值从低优先级队列移至高优先级队列的语句数量 。如果这个数字很快增加，请考虑增加该变量的值。快速增长的计数器意味着优先系统不会使交易过早开始。因为 InnoDB这很可能意味着由于并发事务太多而导致性能下降。

  • STALLED_QUERIES_EXECUTED

    由于执行时间超过thread_pool_stall_limit 系统变量的值而被定义为停止的语句数量 。

  • BECOME_CONSUMER_THREAD

    线程已被分配消费者线程角色的次数。

  • BECOME_RESERVE_THREAD

    线程已被分配预留线程角色的次数。

  • BECOME_WAITING_THREAD

    线程已被分配服务器线程角色的次数。当语句排队时，即使在正常操作中，也会经常发生这种情况，所以在高度负载的系统中，语句排队等候时，此值的快速增加是正常的。

  • WAKE_THREAD_STALL_CHECKER

    停顿检查线程决定唤醒或创建线程以处理某些语句或处理服务器线程角色的次数。

  • SLEEP_WAITS

    THD_WAIT_SLEEP等待 的次数。这些发生在线程进入睡眠状态时; 例如，通过调用该 SLEEP()函数。

  • DISK_IO_WAITS

    THD_WAIT_DISKIO等待 的次数。这些在线程执行可能未命中文件系统缓存的磁盘I / O时发生。当缓冲池读取和写入数据到磁盘时发生这种等待，而不是正常读取和写入文件。

  • ROW_LOCK_WAITS

    THD_WAIT_ROW_LOCK另一个事务等待释放行锁 的数量。

  • GLOBAL_LOCK_WAITS

    数THD_WAIT_GLOBAL_LOCK被释放等待全局锁。

  • META_DATA_LOCK_WAITS

    THD_WAIT_META_DATA_LOCK等待发布元数据锁 的次数。

  • TABLE_LOCK_WAITS

    THD_WAIT_TABLE_LOCK需要访问的待解锁表 的等待数。

  • USER_LOCK_WAITS

    THD_WAIT_USER_LOCK等待由用户线程构造的特殊锁 的数量。

  • BINLOG_WAITS

    数THD_WAIT_BINLOG_WAITS等待二进制日志成为自由。

  • GROUP_COMMIT_WAITS

    THD_WAIT_GROUP_COMMIT等待 的次数。当组织承诺必须等待其他方完成其交易的一部分时才会出现这种情况。

  • FSYNC_WAITS

    数THD_WAIT_SYNC等待文件同步操作。

24.32.3 INFORMATION_SCHEMA TP_THREAD_STATE表

此表为每个由线程池创建的线程处理一个连接。该表具有以下列：

  • TP_GROUP_ID

    线程组ID。

  • TP_THREAD_NUMBER

    线程组中线程的ID。 TP_GROUP_ID并 TP_THREAD_NUMBER一起提供表格中的唯一密钥。

  • PROCESS_COUNT

    使用此线程的语句当前正在执行的10ms间隔。0表示没有语句正在执行，1表示它处于前10ms，等等。

24.33连接控制INFORMATION_SCHEMA表

24.33.1 INFORMATION_SCHEMA CONNECTION_CONTROL_FAILED_LOGIN_ATTEMPTS表

以下各节介绍INFORMATION_SCHEMA与CONNECTION_CONTROL插件关联的 表格 。

24.33.1 INFORMATION_SCHEMA CONNECTION_CONTROL_FAILED_LOGIN_ATTEMPTS表

此表提供有关每个客户端用户/主机组合连续失败连接尝试次数的信息。该表是在MySQL 5.7.17中添加的。

CONNECTION_CONTROL_FAILED_LOGIN_ATTEMPTS 有这些列：

  • USERHOST

    格式中 连接尝试失败的客户端的用户/主机组合 。 'user_name'@'host_name'

  • FAILED_ATTEMPTS

    该值的当前连续失败连接尝试次数USERHOST。这个计算所有失败的尝试，不管它们是否延迟。服务器为其响应添加延迟的尝试次数是该FAILED_ATTEMPTS值与 connection_control_failed_connections_threshold 系统变量值之间的差异 。

备注：

  • CONNECTION_CONTROL_FAILED_LOGIN_ATTEMPTS 必须激活 该 插件以使该表可用，并且CONNECTION_CONTROL必须激活该插件或者表格内容将始终为空。请参见 第6.5.2节“连接控制插件”。

  • 该表仅包含已经有一个或多个连续失败连接尝试而没有后续成功尝试的客户端的行。当客户端连接成功时，其失败连接计数将重置为零，服务器将删除与客户端对应的任何行。

  • connection_control_failed_connections_threshold 在运行时 为系统变量分配值 会将所有累计失败连接计数器重置为零，这会导致表变为空。
