9.1.1字符串文字

字符串是包含在单引号（'）或双引号（"）字符中的字节或字符序列。例子： 

'一串'
“另一个字符串”

被引用的字符串彼此相邻，并连接到单个字符串。以下几行是等同的： 

'一串'
'a''''字符串'

如果ANSI_QUOTES启用了SQL模式，则字符串文字只能在单引号内引用，因为在双引号内引用的字符串被解释为标识符。

甲二进制串是一个字节的字符串。每个二进制字符串都有一个字符集和排序规则binary。一个 非二进制字符串是字符的字符串。它具有除字符集以外的字符集 binary和与字符集兼容的归类。

对于这两种类型的字符串，比较都基于字符串单元的数值。对于二进制字符串，单位是字节; 比较使用数字字节值。对于非二进制字符串，单位是字符，有些字符集支持多字节字符; 比较使用数字字符代码值。字符代码排序是字符串整理的函数。（有关更多信息，请参见 第10.8.5节“二进制排序与_bin排序相比”）。

9.1.2数字文字

数字文字包括精确值（整数）和 DECIMAL文字和近似值（浮点）文字。

整数表示为一系列数字。数字可能包含.小数点分隔符。数字可以在前面-或+分别表示负值或正值。用尾数和指数以科学计数法表示的数字是近似值数字。

精确值数字文字具有整数部分或小数部分，或两者兼有。他们可能会被签名。例如： 1，.2， 3.4，-5， -6.78，+9.10。

近似值数字文字以科学记数法以尾数和指数表示。其中一个或两个部分都可以签名。例如：1.2E3， 1.2E-3，-1.2E3， -1.2E-3。

两个看起来相似的数字可能会有所不同。例如，2.34是一个精确值（定点）数字，而是2.34E0一个近似值（浮点）数字。

日期和时间上下文中的字符串和数字文字。  MySQL DATE以这些格式识别值：

  • 作为任一种'YYYY-MM-DD'或 'YY-MM-DD'格式的字符串。允许使用 “ 宽松 ”的语法：任何标点字符都可以用作日期部分之间的分隔符。例如，'2012-12-31'， '2012/12/31'， '2012^12^31'，和 '2012@12@31'是相等的。

  • 作为不带任何分隔符 'YYYYMMDD'或'YYMMDD' 字符串的字符串，前提是该字符串作为日期有意义。例如，'20070523'与 '070523'被解释为 '2007-05-23'，但是 '071332'是非法的（它有无意义的月和日），成为 '0000-00-00'。

  • 作为数字YYYYMMDD或 YYMMDD格式的数字，只要该数字作为日期有意义。例如， 19830905并且830905 被解释为'1983-09-05'。

MySQL 以这些格式识别DATETIME和 TIMESTAMP赋值：

  • 作为任一种'YYYY-MM-DD HH:MM:SS'或'YY-MM-DD HH:MM:SS'格式的字符串。这里也允许使用“ 宽松 ”的语法：任何标点字符都可以用作日期部分或时间部分之间的分隔符。例如，'2012-12-31 11:30:45'， '2012^12^31 11+30+45'， '2012/12/31 11*30*45'，和 '2012@12@31 11^30^45'是相等的。

    日期和时间部分和小数部分之间唯一标识的分隔符是小数点。

    日期和时间部分可以用T空格分隔 而不用空格。例如， '2012-12-31 11:30:45' '2012-12-31T11:30:45'是等同的。

  • 作为不带任何分隔符 'YYYYMMDDHHMMSS'或 'YYMMDDHHMMSS'字符串的字符串，前提是该字符串作为日期有意义。例如， '20070523091528'并且 '070523091528'被解释为 '2007-05-23 09:15:28'，但 '071122129015'是非法的（它有一个无意义的分钟部分）和变'0000-00-00 00:00:00'。

  • 作为数字YYYYMMDDHHMMSS或 YYMMDDHHMMSS格式的数字，只要该数字作为日期有意义。例如， 19830905132800并且 830905132800被解释为 '1983-09-05 13:28:00'。

A DATETIME或 TIMESTAMP值可以包含最大微秒（6位）精度的尾随小数秒部分。小数部分应始终与其余时间分开小数点; 没有其他小数分隔符被识别。有关MySQL 中小数秒支持的信息，请参见第11.3.6节“小数秒的 时间值”。

包含两位数年份值的日期含糊不清，因为该世纪未知。MySQL使用这些规则解释两位数年份值：

  • 范围内的年份值70-99将转换为1970-1999。

  • 范围内的年份值00-69将转换为2000-2069。

另见第11.3.8节“日期中的两位数字”。

对于指定为包含日期部分分隔符的字符串的值，没有必要为小于的月份或日期值指定两位数字10。 '2015-6-9'是一样的 '2015-06-09'。同样，对于指定为包含时间部分分隔符的字符串的值，无需为小于，小于，小于或小于秒的小时，分​​钟或秒值指定两位数字10。'2015-10-30 1:2:3'是一样的'2015-10-30 01:02:03'。

指定为数字的值应该是6,8,12或14位数字。如果一个数字的长度是8或14位数，那么它被假定为 YYYYMMDD或YYYYMMDDHHMMSS 格式，并且该年份由前4位数字给出。如果数字长度为6或12位数字，则假定为数字 YYMMDD或YYMMDDHHMMSS 格式，并且年份由前两位数字给出。不是这些长度之一的数字被解释为用前导零填充到最近的长度。

指定为非限定字符串的值根据其长度进行解释。对于长度为8或14个字符的字符串，假定年份由前4个字符给出。否则，假定年份由前2个字符给出。该字符串从左到右被解释为查找年份，月份，日期，小时，分钟和秒值，以查找字符串中存在的部分。这意味着你不应该使用少于6个字符的字符串。例如，如果你指定'9903'，认为代表1999年3月，MySQL将它转换为“ 零 ”日期值。这是因为年份和月份值 99和03，但是那一天却完全没了。但是，您可以明确指定零值来表示丢失的月份或日期部分。例如，要插入值'1999-03-00'，请使用'990300'。

MySQL TIME以这些格式识别值：

  • 作为'D HH:MM:SS'格式的字符串。您也可以使用以下的一个“ 宽松 ” 的语法：'HH:MM:SS'， 'HH:MM'，'D HH:MM'， 'D HH'，或'SS'。这里 D代表日子，可以有一个从0到34的值。

  • 作为一个没有分隔符'HHMMSS'格式的字符串 ，只要它有意义的时间。例如，'101112'被理解为'10:11:12'，但是 '109712'是非法的（它具有无意义的微小部分）并且变成了'00:00:00'。

  • 作为一种HHMMSS格式的数字，只要它有时间意义。例如， 101112被理解为 '10:11:12'。以下备选格式也理解：SS， MMSS，或HHMMSS。

结尾的部分的秒部分在所识别的 'D HH:MM:SS.fraction'， 'HH:MM:SS.fraction'， 'HHMMSS.fraction'，和 HHMMSS.fraction时间的格式，其中， fraction是在高达微秒（6位）精度的小数部分。小数部分应始终与其余时间分开小数点; 没有其他小数分隔符被识别。有关MySQL 中小数秒支持的信息，请参见第11.3.6节“小数秒的 时间值”。

对于TIME指定为包含时间部分分隔符的字符串的值，无需为小于，小于，小于，小于或等于秒的小时，分​​钟或秒值指定两位数字10。'8:3:2' 是一样的'08:03:02'。

9.2.1标识符限定符

对象名称可能不合格或不合格。在名称的解释是明确的上下文中允许使用非限定名称。限定名称至少包含一个限定词，通过覆盖默认上下文或提供缺少的上下文来阐明解释上下文。

例如，此语句使用非限定名称创建一个表t1： 

CREATE TABLE t1（i INT）;

由于不t1包含用于指定数据库的限定符，因此语句将在默认数据库中创建表。如果没有默认数据库，则会发生错误。

该语句使用限定名称创建一个表 db1.t1： 

CREATE TABLE db1.t1（i INT）;

由于db1.t1包含数据库限定符 db1，因此无论默认数据库如何，语句都会t1在名为的数据库中 创建 db1。如果没有默认数据库，则必须指定限定符。限定符可以，如果有一个默认的数据库中指定，指定不同于默认数据库，或使数据库明确如果默认是一样的指定的一个。

限定符具有以下特征：

  • 一个非限定名称由一个标识符组成。一个限定名称由多个标识符组成。

  • 多部分名称的组成部分必须用句点（.）字符分隔。多部分名称的初始部分充当限定符，影响解释最终标识符的上下文。

  • 限定符是一个单独的标记，不需要与关联的标识符连续。例如， tbl_name.col_name并且 tbl_name . col_name是等同的。

  • 如果多部分名称的任何组件需要引用，请单独引用它们而不是引用整个名称。例如，写 `my-table`.`my-column`，而不是 `my-table.my-column`。

  • 在限定名称后面跟随一个句点的保留字必须是一个标识符，因此在该上下文中不需要引用它。

  • 该语法 .tbl_name 表示tbl_name默认数据库中的表格。

    注意

    这个语法从MySQL 5.7.20开始不推荐使用，并且将在未来版本的MySQL中删除。

  • 数据库名称是完全限定的并且不具有限定符： 

    CREATE DATABASE db1;

    CREATE TRIGGER mytrigger ... ON mytable ...;

CREATE TRIGGER mytrigger ... ON mydb.mytable ...;

9.2.2标识符大小写敏感性

在MySQL中，数据库对应于数据目录中的目录。数据库中的每个表对应于数据库目录中的至少一个文件（可能更多，取决于存储引擎）。触发器也对应于文件。因此，底层操作系统的区分大小写在数据库，表和触发器名称的区分大小写中起着一定的作用。这意味着这些名称在Windows中不区分大小写，但在大多数Unix中都区分大小写。一个明显的例外是OS X，它基于Unix，但使用不区分大小写的默认文件系统类型（HFS +）。但是，OS X也支持UFS卷，与任何Unix一样区分大小写。看到 第1.8.1节“MySQL标准SQL扩展”。所述 lower_case_table_names系统变量也影响服务器如何处理标识符区分大小写，如在本节后面说明。

注意

虽然数据库，表和触发器名称在某些平台上不区分大小写，但不应在同一语句中使用不同的情况引用其中的一种。以下语句不起作用，因为它引用了一个表格my_table，如下所示 MY_TABLE： 

MySQL的> SELECT * FROM my_table WHERE MY_TABLE.col=1;

列，索引，存储例程和事件名称在任何平台上都不区分大小写，列别名也不区分大小写。

但是，日志文件组的名称区分大小写。这与标准SQL不同。

默认情况下，表别名在Unix上是区分大小写的，但在Windows或OS X上不是这样。以下语句在Unix上不起作用，因为它引用了别名as a和as A： 

mysql> 
    - >SELECT col_name FROM tbl_name AS aWHERE a.col_name = 1 OR A.col_name = 2;

但是，Windows上也允许使用相同的语句。为了避免这些差异导致的问题，最好采用一致的约定，例如始终使用小写名称创建和引用数据库和表。建议使用此惯例以实现最大的便携性和易用性。

表和数据库名称如何存储在磁盘上并在MySQL中使用受lower_case_table_names系统变量影响， 系统变量可以在启动mysqld时设置 。 lower_case_table_names可以采用下表中显示的值。这个变量不 不影响触发标识符的情况下的灵敏度。在Unix上，默认值为 lower_case_table_names0.在Windows上，默认值为1.在OS X上，默认值为2。

9.2.3标识符到文件名称的映射

数据库和表标识符与文件系统中的名称之间存在对应关系。对于基本结构，MySQL将数据库中的每个数据库表示为数据目录中的一个目录，每个表由相应数据库目录中的一个或多个文件组成。对于表格格式文件（.FRM），数据总是存储在此结构和位置中。

对于数据和索引文件，磁盘上的确切表示是存储引擎特定的。这些文件可能存储在与FRM文件相同的位置，或者信息可能存储在单独的文件中。 InnoDB数据存储在InnoDB数据文件中。如果您使用表空间InnoDB，则使用 您创建的特定表空间文件。

内置函数名称解析

解析器使用默认规则来分析内置函数的名称。这些规则可以通过启用IGNORE_SPACESQL模式进行更改 。

当分析器遇到一个单词是内置函数的名称时，它必须确定名称是表示函数调用，还是表示对表名或列名等标识符的非表达式引用。例如，在以下语句中，第一个引用 count是函数调用，而第二个引用是表名： 

SELECT COUNT（*）FROM mytable;
CREATE TABLE count（i INT）;

解析器应该只在解析期望的表达式时才将内建函数的名称识别为指示函数调用。也就是说，在非表达式上下文中，函数名称被允许作为标识符。

但是，某些内置函数具有特殊的解析或实现注意事项，因此解析器默认使用以下规则来区分它们的名称是用作函数调用还是用作非表达式上下文中的标识符：

  • 要将该名称用作表达式中的函数调用，名称和下列(括号字符之间不得有空格 。

  • 相反，要使用函数名称作为标识符，不能立即使用括号。

  • 用户定义的功能和存储的功能

9.3关键字和保留字

关键字是在SQL中有重要意义的词。某些关键字，如SELECT， DELETE或 BIGINT，被保留，需要用作标识符，例如表和列名特殊待遇。对于内置函数的名称也是如此。

非保留的关键字被允许作为标识符而不用引用。如果您按照第9.2节“模式对象名称”中的说明引用保留字，则允许保留字作为标识符： 

MySQL的> CREATE TABLE interval (begin INT, end INT);
错误1064（42000）：您的SQL语法中有错误...
接近'间隔（开始INT，结束INT）'

BEGIN并且END是关键字但不是保留的，所以它们作为标识符的使用不需要引用。INTERVAL是一个保留关键字，必须用引号将其用作标识符： 

MySQL的> CREATE TABLE `interval` (begin INT, end INT);
查询OK，0行受影响（0.01秒）

例外：在限定名称后面跟随一个句点的单词必须是一个标识符，因此即使它被保留也不需要引用它： 

MySQL的> CREATE TABLE mydb.interval (begin INT, end INT);
查询OK，0行受影响（0.01秒）

内置函数的名称被允许作为标识符，但可能需要谨慎使用。例如， COUNT可以作为列名接受。但是，默认情况下，在函数名称和以下( 字符之间的函数调用中不允许有空格。该要求使解析器能够区分该名称是在函数调用还是在非函数环境中使用。有关识别函数名称的更多详细信息，请参见 第9.2.4节“函数名称解析和解析”。

9.4用户定义的变量

您可以在一个语句中将值存储在用户定义的变量中，稍后在另一个语句中引用它。这使您可以将值从一个语句传递到另一个语句。

用户变量被写成 ，其中变量名组成的字母数字字符，， ，和。如果引用它作为一个字符串或标识符（例如，用户的变量名可以包含其他字符， 或）。 @var_namevar_name._$@'my-var'@"my-var"@`my-var`

用户定义的变量是特定于会话的。由一个客户端定义的用户变量不能被其他客户端看到或使用。（例外：有权访问Performance Schema user_variables_by_thread表的用户可以看到所有会话的所有用户变量。）当客户端退出时，给定客户机会话的所有变量都会自动释放。

用户变量名称不区分大小写。名称的最大长度为64个字符。

一种设置用户定义变量的方法是发出一个 SET 声明： 

SET @ var_name= expr[，@ var_name= expr] ...

因为SET，=或者 :=可以用作赋值运算符。

您也可以使用除以外的语句为用户变量赋值SET。在这种情况下，赋值运算符必须是， :=而不是 =因为后者=在非SET 语句中被视为比较运算符 ： 

mysql> SET @t1=1, @t2=2, @t3:=4;
mysql>SELECT @t1, @t2, @t3, @t4 := @t1+@t2+@t3;
+ ------ + ------ + ------ + -------------------- +
| @ t1 | @ t2 | @ t3 | @ t4：= @ t1 + @ t2 + @ t3 |
+ ------ + ------ + ------ + -------------------- +
| 1 | 2 | 4 | 7 |
+ ------ + ------ + ------ + -------------------- +

用户变量可以从一组有限的数据类型中分配一个值：整型，小数，浮点型，二进制或非二进制字符串或NULL值。小数和实数值的赋值不会保留数值的精度或比例。除一种允许类型以外的类型的值被转换为允许类型。例如，具有时间或空间数据类型的值被转换为二进制字符串。具有该JSON数据类型的值将转换为字符集utf8mb4和 字符集的字符串 utf8mb4_bin。

如果为用户变量分配了非二进制（字符）字符串值，则它与字符串具有相同的字符集和归类。用户变量的强制性是隐含的。（这与表列值相同。）

分配给用户变量的十六进制或位值被视为二进制字符串。要将十六进制或位值作为数字分配给用户变量，请在数字上下文中使用它。例如，添加0或使用CAST(... AS UNSIGNED)： 

mysql> SET @v1 = X'41';
mysql> SET @v2 = X'41'+0;
mysql> SET @v3 = CAST(X'41' AS UNSIGNED);
mysql>SELECT @v1, @v2, @v3;
+ ------ + ------ + ------ +
| @ v1 | @ v2 | @ v3 |
+ ------ + ------ + ------ +
| A | 65 | 65 |
+ ------ + ------ + ------ +
mysql> SET @v1 = b'1000001';
mysql> SET @v2 = b'1000001'+0;
mysql> SET @v3 = CAST(b'1000001' AS UNSIGNED);
mysql>SELECT @v1, @v2, @v3;
+ ------ + ------ + ------ +
| @ v1 | @ v2 | @ v3 |
+ ------ + ------ + ------ +
| A | 65 | 65 |
+ ------ + ------ + ------ +

如果在结果集中选择了用户变量的值，则它将作为字符串返回给客户端。

如果您引用了一个尚未初始化的变量，则它具有一个值NULL和一个字符串类型。

用户变量可用于允许表达式的大多数情况下。这目前不包括明确要求文字值的上下文，如在 LIMIT一个条款 SELECT声明，或 一条款声明。 IGNORE N LINESLOAD DATA

作为一般规则，除了在 SET 语句中，您不应该为用户变量赋值并在同一语句中读取该值。例如，要增加一个变量，这是可以的： 

SET @a = @a + 1;

对于其他陈述，例如 SELECT，您可能会得到您期望的结果，但这不能保证。在下面的语句中，你可能会认为MySQL会@a先评估 ，然后再做一个任务： 

SELECT @a，@a：= @ a + 1，...;

但是，涉及用户变量的表达式的评估顺序未定义。

将值赋给变量并读取同一非SET 语句中的值的另一个问题是变量的默认结果类型是基于语句开头的类型。以下示例说明了这一点： 

mysql> SET @a='test';
mysql>SELECT @a,(@a:=20) FROM tbl_name;

对于这条SELECT语句，MySQL向客户报告第一列是一个字符串，并将所有访问转换@a为字符串，即使@a被设置为第二行的数字。在之后 SELECT的语句执行， @a被视为一个号码下一条语句。

为了避免这种行为的问题，要么不分配一个值，并在一个语句中读取同一变量的值，否则设置变量0， 0.0或者''你使用它之前确定其类型。

在SELECT声明中，每个选择表达式仅在发送给客户端时才被评估。这意味着，在HAVING，GROUP BY或ORDER BY条款，指的是在选择表达式列表分配一个值不变量 不按预期方式工作： 

MySQL的> SELECT (@aa:=id) AS a, (@aa+3) AS b FROM tbl_name HAVING b=5;

到基准b中 HAVING子句是指一个别名在使用选择列表中的表达式@aa。这不符合预期：@aa包含id来自上一个选定行的值，而不是来自当前行的值。

用户变量旨在提供数据值。它们不能在SQL语句中直接作为标识符或作为标识符的一部分使用，例如在需要表或数据库名称的上下文中，或者作为保留字（如 SELECT。即使变量被引用也是如此，如以下示例所示：

9.6注释语法

MySQL服务器支持三种评论风格：

  • 从#角色到行尾。

  • 从一个-- 序列到行尾。在MySQL中，-- （double-dash）注释样式需要在第二个短划线之后加上至少一个空格或控制字符（例如空格，制表符，换行符等）。这个语法与标准的SQL注释语法稍有不同，如 第1.8.2.4节“作为注释开始的'''所述。

  • 从一个/*序列到下面的 */序列，就像在C编程语言中一样。该语法使得注释可以在多行上延伸，因为开始和结束序列不必在同一行上。

以下示例演示了所有三种评论样式： 

mysql> SELECT 1+1;     # This comment continues to the end of line
mysql> SELECT 1+1;     -- This comment continues to the end of line
mysql> SELECT 1 /* this is an in-line comment */ + 1;
mysql>SELECT 1+
/*
this is a
multiple-line comment
*/
1;

不支持嵌套注释。（在某些情况下，嵌套评论可能被允许，但通常不是，用户应该避免它们。）

MySQL服务器支持一些C风格注释的变体。这些使您可以使用以下格式的注释编写包含MySQL扩展的代码，但仍然可以移植： 

/ *！MySQL-specific code* /

在这种情况下，MySQL服务器解析并执行注释中的代码，就像其他任何SQL语句一样，但其他SQL服务器将忽略扩展。例如，MySQL服务器STRAIGHT_JOIN在以下语句中识别关键字，但其他服务器不会： 

选择 /*！STRAIGHT_JOIN * / col1 FROM table1，table2 WHERE ...

如果在! 字符后添加版本号，则仅当MySQL版本大于或等于指定的版本号时才执行注释中的语法。KEY_BLOCK_SIZE以下注释中的关键字仅由MySQL 5.1.10或更高版本的服务器执行： 

CREATE TABLE t1（INT，KEY（a））/ *！50110 KEY_BLOCK_SIZE = 1024 * /;

刚刚描述的注释语法适用于mysqld服务器如何 分析SQL语句。在 MySQL的客户端程序将它们发送到服务器之前还执行语句的一些分析。（这是为了确定多语句输入行内的语句边界。）

这种格式的注释/*!12345 ... */不存储在服务器上。如果此格式用于评论存储例程，则注释将不会保留在服务器上。

C风格评论语法的另一种变体用于指定优化器提示。提示注释包括注释打开序列+ 后面的字符/*。例： 

SELECT / * + BKA（t1）* / FROM ...;

有关更多信息，请参见第8.9.2节“优化器提示”。
